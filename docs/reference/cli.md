# CLI reference

robot-arm-sim provides three subcommands corresponding to the three pipeline
stages. Run any command with `--help` for built-in usage.

## Global options

```
robot-arm-sim [OPTIONS] COMMAND [ARGS]
```

| Option | Description |
|--------|-------------|
| `--version`, `-v` | Show version and exit |
| `--install-completion` | Install shell completion for the current shell |
| `--show-completion` | Show shell completion script |
| `--help` | Show help and exit |

---

## `analyze`

Analyze STL files and generate per-part YAML analysis + renders.

```bash
$ robot-arm-sim analyze ROBOT_DIR [--override-manual]
```

| Argument / Option | Description |
|-------------------|-------------|
| `ROBOT_DIR` | Path to robot directory. Must contain a `stl_files/` subdirectory with one STL file per link. |
| `--override-manual` | Re-detect all connection points, discarding manual placements. |

**Output:** creates an `analysis/` directory inside `ROBOT_DIR` with:

- One YAML file per part (geometry, connection points, features)
- A `summary.yaml` listing all parts with role hints
- Multi-angle renders of each part

**Example:**

```bash
$ robot-arm-sim analyze robots/Meca500-R3/
```

---

## `generate`

Generate a URDF file from a kinematic chain specification and analysis data.

```bash
$ robot-arm-sim generate ROBOT_DIR [CHAIN_FILE] [--output PATH]
```

| Argument / Option | Description |
|-------------------|-------------|
| `ROBOT_DIR` | Path to robot directory. Must contain an `analysis/` subdirectory. |
| `CHAIN_FILE` | Path to the `chain.yaml` kinematic chain specification. Default: `<ROBOT_DIR>/chain.yaml`. |
| `--output PATH` | Output URDF path. Default: `<ROBOT_DIR>/robot.urdf`. |

**Example:**

```bash
$ robot-arm-sim generate robots/Meca500-R3/
```

```bash
$ robot-arm-sim generate robots/UR5/ robots/UR5/chain.yaml --output /tmp/ur5.urdf
```

---

## `simulate`

Launch the interactive 3D simulator in the browser.

```bash
$ robot-arm-sim simulate ROBOTS_DIR [--port PORT]
```

| Argument / Option | Description |
|-------------------|-------------|
| `ROBOTS_DIR` | Path to a single robot folder or a directory containing robot folders (each with a `robot.urdf`). |
| `--port PORT` | Port for the web UI. Default: `8080`. |

The simulator auto-discovers all robots under `ROBOTS_DIR` that contain a
`robot.urdf` file. If multiple robots are found, a dropdown selector appears
in the UI.

To choose which robot is selected by default, create a `.default` file in
`ROBOTS_DIR` containing the robot folder name (e.g. `Meca500-R3`). If the
file is absent or names an unknown robot, the first robot alphabetically is
used.

**Example:**

```bash
$ robot-arm-sim simulate robots/Meca500-R3/
```

```bash
$ robot-arm-sim simulate robots/ --port 9090
```

The simulator serves the web UI at `http://localhost:<PORT>`. STL files are served at `/stl/<robot_name>/` paths.
A health-check endpoint is available at `/healthz`.

## Environment variables

These variables affect the `simulate` command at runtime.

| Variable | Description |
|----------|-------------|
| `CLUSTER_MODE` | Set to `1` to hide the **Stop Simulator** button. Useful when the process lifecycle is managed externally. |
| `KUBERNETES_SERVICE_HOST` | Automatically set by Kubernetes. When present, cluster mode is enabled implicitly (same effect as `CLUSTER_MODE=1`). |

Either variable being set is sufficient; they are OR-ed together.

---

## HTTP endpoints

Endpoints served by `simulate` on the configured port.

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/` | Main UI. Loads the default robot. |
| `GET` | `/{robot_name}` | UI for a specific robot. Returns an error label if the name is unknown. |
| `GET` | `/healthz` | Health check. Returns `{"status": "ok"}`. |
| `GET` | `/stl/{robot_name}/*` | Static STL mesh files from the robot's `stl_files/` directory. |

The Docker image uses `ENTRYPOINT ["robot-arm-sim"]` and
`CMD ["simulate", "/app/robots"]`, so the container exposes these endpoints on
port 8080 by default.

## Robot directory layout

Each robot lives in its own directory under the `ROBOTS_DIR` passed to
`simulate` (or to the pipeline commands individually). The expected structure:

```
ROBOTS_DIR/
  .default              # optional — single line naming the default robot folder
  RobotName/
    stl_files/          # one STL per link (required by analyze)
    analysis/           # generated by analyze (YAML + renders per part)
    chain.yaml          # kinematic chain specification (input to generate)
    robot.urdf          # generated by generate (required by simulate)
```

`simulate` auto-discovers every subdirectory of `ROBOTS_DIR` that contains a
`robot.urdf`. If `ROBOTS_DIR` itself contains `robot.urdf`, it is treated as a
single-robot directory.

The `.default` file is a plain-text file containing the folder name of the
robot to select on first load (e.g. `Meca500-R3`). If the file is absent or
names an unknown robot, the first robot alphabetically is used.
